    Programming Language Concepts⇦

    University course

    These notes were made for the University of Southampton COMP2212 module.

    Theory⇦

    Lexing and Parsing

    Grammars can be used to create valid program strings. Lexing and parsing takes the opposite approach by checking if there is a derivation for a given string. This process generally consists of:

    • Scanning program strings
    • Identifying tokens using regular expressions (lexing)
    • Producing abstract syntax (parsing)
    Lexing

    Lexing is the process of taking a string and converting it into a set of tokens. These tokens are the building blocks of the program and can be thought of as words or punctuation.

    Tokens are generaly identified by using regular expressions.

    Parsing

    Parsing is the process of taking tokens and converting them into an abstract syntax tree.

    A top down parser starts at the root/beginning (generally start/entry point of code) and recursively explores required tokens. The most common method for doing this is recursive descent. A quick example would be if it found an addition operation. It would add the operation to the syntax tree and then explore the operands.

    A bottom up parser takes the opposite approach and builds up the syntax tree from the leaves first.

    Types

    Types are a particular classification of program behaviours. More generally types are labels for the sturcture of data we expect to be manipulating.

    Types are both abstract descriptions of programs (we can study correctness of properties by abstracting away lower level details) as well as precise descriptions of programs (we can use mathematical tools to formalise and check properties).

    • Strongly typed prevents acess to private members, corruption of memory etc
    • Weakly typed less restrictive and generally used for semantics
    • Static typing type checking that is done at compile time
    • Dynamic typing type checking which is done during run time

    Coursework⇦

    Required features

    The following is a list of features which must be included for this language to be useful.

    • Accept finite sequences of integers as streams
    • Accept multiple streams
    • Allow general non-terminal stream operations
    • Allow operations which accept (combine) multiple streams
    • Allow global variables (not bound to a stream)
    • Allow recursive operations
    Restrictions

    Using the features required above and some insight into the spec we can make the following restrictions.

    • Streams must be fully independent
    • Input Streams must be immutable
    General design ideas

    Read in a stream with 

    LoadStreams "Inputfile.txt";

    That will read in n streams and define the globals InStream0, InStream1, InStream2, ... InStreamN as well as NumInStreams.

    For output purposes the debug operation Show will print a stream out and pass it on.

    LoadStreams "Inputfile.txt";
InStream0 -> Show;
InStream1 -> Show; 
NumInStreams -> Show;

    Comments can be added with the use of #

    # input #
# 1 2   #
# 5 6   #
# 1 5   #

LoadStreams "Inputfile.txt"; #Load in streams
InStream0 -> Show;    # 1 5 1
InStream1 -> Show;    # 2 6 5
NumInStreams -> Show; # 2

    Output streams are also defined in a similar way, OutStream0, OutStream1, OutStream2 ... OutStreamN as well as NumOutStreams. Operations can target these. For example copy which simply accepts an input and passes it on with no modifications.

    InStream0 -> Copy -> OutStream0;

    We can combine multiple operations to form more complex behaviours. For example we can show the elements we are copying.

    # input #
# 1 2   #
# 5 6   #
# 1 5   #

LoadStreams "Inputfile.txt"; #Load in streams

InStream0 -> Show -> Copy -> OutStream0; #Prints 1 5 1

    Operations can accept multiple streams to reduce them. An example of this would be an add operation.

    # input #
# 1 2   #
# 5 6   #
# 1 5   #

LoadStreams "Inputfile.txt"; #Load in streams

InStream0|InStream1 -> Add -> Show -> OutStream0; #Prints 3 11 6

    Example solutions for the problems could be made this way

    Problem1
    LoadStreams "Inputfile.txt";

Gen 0 -> OutStream0;
InStream0 -> Copy -> OutStream0;

OutputStreams 1;
    Problem2
    LoadStreams "Inputfile.txt";

InStream0 -> Copy -> OutStream0;
InStream0 -> Copy -> OutStream1;

OutputStreams 2;
    Problem3
    LoadStreams "Inputfile.txt";

InStream0 | (InStream1 -> Multiply 3) -> Add -> OutStream0;

OutputStreams 1;
    Problem4
    LoadStreams "Inputfile.txt"

InStream0 -> Previous (Add) -> OutStream0;

OutputStreams 1;
